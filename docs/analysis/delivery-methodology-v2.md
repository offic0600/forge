# Forge 交付方法论 v2.0

> 基于 32 个 Session / 9 天 / 8 个 Phase 的实战提炼
> 日期: 2026-02-24 | 数据来源: dev-logbook.md + buglist.md + 基线文档

---

## 一、核心理念

> **用文档对抗 AI 遗忘，用验收测试对抗质量腐化，用双基线对抗设计偏移，用知识沉淀对抗经验流失。**

传统软件交付的核心约束是"人的产能"，AI 协同交付的核心约束变成了**"上下文的连续性"和"质量的可控性"**。模型可以每小时产出 2,000+ 行代码，但如果没有系统性的记忆、验证、沉淀机制，高产出只会加速混乱。

本方法论解决的不是"怎么让 AI 写代码"，而是**"怎么让人 + AI 在多 Session、多 Phase 的长周期项目中持续、可控地交付高质量软件"**。

## 二、交付模型

### 2.1 三层迭代结构

```
┌─────────────────────────────────────────────────┐
│  Phase（里程碑）                                  │
│  ├─ 粒度：1 个可验收的功能集                       │
│  ├─ 周期：1-6 个 Session（弹性）                   │
│  ├─ 产出：可运行的系统 + 验收测试 + 基线更新         │
│  └─ 闭环：验收 → 基线交叉校验 → 经验编码            │
│                                                   │
│  Session（最小工作单元）                            │
│  ├─ 粒度：1 次 Claude Code 对话（~2-3 小时）        │
│  ├─ 周期：固定 PDCA 微循环                         │
│  ├─ 产出：代码 + 测试 + logbook 条目 + Git commit   │
│  └─ 闭环：文件变更表 → 经验沉淀 → 统计快照          │
│                                                   │
│  Turn（执行回合）                                   │
│  ├─ 粒度：1 次 Agentic Loop 迭代                   │
│  ├─ 周期：秒级                                     │
│  ├─ 产出：工具调用结果 / 代码片段                    │
│  └─ 闭环：OODA → 底线检查 → 失败自修复              │
└─────────────────────────────────────────────────┘
```

**实证数据**:

| 维度 | 数值 |
|------|------|
| 总 Phase 数 | 8 个（Phase 0-7） |
| 总 Session 数 | 32 |
| 平均 Session/Phase | 4（范围 1-6） |
| 平均 Session 时长 | ~2 小时 |
| 平均代码产出/Session | ~1,700 行 |

### 2.2 Session 微型 PDCA 循环

每个 Session 遵循固定结构，**不允许跳步**：

```
Plan  → 声明本 Session 目标（1 句话）
Do    → 实施（编码 + 测试 + Bug 修复）
Check → 文件变更表 + 统计快照 + 运行验证
Act   → 经验沉淀 + Git commit + logbook 更新
```

**为什么是 PDCA 而不是自由发挥**：Session 5（Docker 部署）用 90 分钟、13 次尝试解决 10 个问题，但因为没有结构化记录，Session 14 重建时又踩了 3 个相同的坑。从 Session 15 建立 PDCA 纪律后，同类问题零复发。

### 2.3 Phase 命名与编号规范

| 规则 | 说明 | 反面教训 |
|------|------|---------|
| 格式 `Phase X.Y — 语义标题` | 如 `Phase 1.6 — AI 交付闭环` | — |
| 新增中间阶段用小数点扩展 | Phase 1 和 Phase 2 之间插入 1.5、1.6 | — |
| **禁止重命名已有编号** | 编号一旦确定，永不更改 | Session 13: Phase 2.5→1.6 导致 69 处全局替换，耗时 2 小时 |

## 三、质量保障体系

### 3.1 四道防线

```
第 1 道：本地快速验证（30 秒反馈）
  └─ 单元测试 + npm run build + 编译检查

第 2 道：底线自动检查（Turn 级）
  └─ code-style + security baseline，失败自动修复（最多 2 轮）

第 3 道：验收测试（Phase 级）
  └─ 结构化 TC，自动化 curl + 手动 UI，通过率 ≥ 90%

第 4 道：双基线交叉校验（Phase 级）
  └─ 设计基线 vs 规划基线 vs 代码实现，三角校验
```

**实证：四道防线的拦截率**

| 防线 | 拦截的典型问题 | Session 示例 |
|------|--------------|-------------|
| 本地快速验证 | TypeScript 类型错误、Kotlin 编译失败 | S15: npm run build 拦截了 3 个类型错误 |
| 底线自动检查 | CORS 通配符、硬编码凭证 | S22: baseline auto-check 首次干净通过 |
| 验收测试 | workspace 工具 REST API 未连通 | S14: 单元测试全过 ≠ 产品可用 |
| 双基线校验 | 规划说做了但设计基线没体现 | S13: 发现 15 处格式/逻辑不一致 |

### 3.2 验收测试驱动（ATDD）

**生命周期**：

```
Phase 编码前    Phase 编码中    Phase 编码后    Phase 验收
    │              │              │              │
    ▼              ▼              ▼              ▼
 写场景标题    填充 TC 细节    交叉验证校准    运行 + 回填数据
 + 关键预期    + 操作步骤     + 数据校正      + 统计通过率
```

**关键原则**：
- **场景先行**：编码前写场景标题 + 预期（Session 13 教训：后补验收测试发现 16 处错误）
- **交叉验证**：TC 的操作和预期必须与代码实现逐条比对
- **数据校准**：实际运行后回填真实数据（Profile 名称、工具数量、端点数等）

**统计数据**：

| Phase | TC 数 | 通过率 | 说明 |
|-------|-------|--------|------|
| Phase 1.6 | 87 | 92.0% | 21 场景，80/87 通过 |
| Sprint 2.1 | 34 | 100% | CI/Playwright/Bug 修复 |
| Sprint 2.2 | 24 | 100% | MCP 真实服务连接 |
| Phase 3 | 24 | 83.3% | 首轮，后修复到 100% |
| Phase 5 | 38 | 94.7% | 记忆 + 压缩 |
| Phase 6 | 11 | 100% | Git 载入 + 知识 Scope |

### 3.3 Bug 治理模型

**32 个 Bug 的根因分析**：

| 根因类型 | 数量 | 占比 | 典型表现 | 防御手段 |
|---------|------|------|---------|---------|
| 枚举序列化 | 3 | 9% | Kotlin 枚举大写 vs 前端期望小写 | 新增枚举立即全局 grep |
| 多入口参数遗漏 | 4 | 13% | REST API 和 WebSocket 参数不同步 | 同一入口多路径测试 |
| Token/Context 溢出 | 3 | 9% | System Prompt 过大、WebSocket 消息超限 | 渐进式加载 + 压缩 |
| Docker 环境差异 | 5 | 16% | Alpine 缺依赖、TLS 失败、JDK 版本 | environment-parity Skill |
| 空字符串 vs null | 2 | 6% | `q=` vs 不传 `q` 处理不同 | isNullOrBlank() |
| UI 交互 | 8 | 25% | 按钮不可见、焦点丢失、文件树不刷新 | 验收测试 UI 场景 |
| 其他 | 7 | 22% | — | — |

**系统性 Bug 的全局排查纪律**：

发现一个 Bug 属于某类系统性问题时（如枚举序列化），**立即**全局排查同类问题并一次性修复：

```bash
# 示例：发现 BUG-008 (FileType 枚举)
# 立即执行：
grep -r "enum class" --include="*.kt"
# → 找到 3 处 → 一次性全部加 @JsonValue → 零复发
```

Session 15 发现 12 个 Bug（占 Phase 1.6 总 Bug 的 60%），正是因为这是第一次系统性测试 Session。**教训：越早做系统性测试，越早发现聚集性 Bug**。

## 四、知识管理体系

### 4.1 经验编码管道

```
发现经验
    │
    ▼
logbook 经验沉淀（即时记录）
    │
    ▼
验证（跨 2+ Session 确认有效）
    │
    ▼
编码到 CLAUDE.md（已知陷阱 / 纪律 / 方法论）
    │
    ▼
进一步固化为 Skill / Baseline（平台能力）
```

**实证**：CLAUDE.md 从 Session 1 的 67 行增长到 243 行，每一条都是实战验证过的知识：

| 知识类型 | 条目数 | 来源 |
|---------|--------|------|
| 已知陷阱 | 9 条 | 多 Session 重复踩坑后编码 |
| 开发纪律 | 4 大支柱 | Session 13/15/18 提炼 |
| 交付方法论 | 5 个实践 | Session 18 方法论分析 |
| Docker 注意事项 | 7 条 | Session 5/14/19/20 累积 |
| MCP 工具清单 | 17 条 | 实时校准 |

**知识密度增长曲线**：

```
Session  1-5:   67 行 → 67 行    （无沉淀意识）
Session  5-10:  67 行 → 67 行    （痛点在累积但未编码）
Session 10-15:  67 行 → 181 行   （首次系统性编码，建立纪律）
Session 15-25: 181 行 → 220 行   （稳定增量）
Session 25-32: 220 行 → 243 行   （趋于稳定，新知识通过 Skill 沉淀）
```

**关键洞察**：Session 10 之前的知识全靠人脑记忆，导致同类问题反复出现。Session 15 建立编码纪律后，知识开始系统性积累，Bug 复发率降为零。

### 4.2 文档债务控制

| 规则 | 频率 | 说明 |
|------|------|------|
| logbook 每 Session 更新 | 每次 | 不更新不提交 |
| 基线交叉校验 | 每 Phase 结束 | 设计 vs 规划 vs 代码三角校验 |
| 文档全量审查 | 每 5 个 Session | 格式统一、去重、清理版本批注 |
| 统计数据校准 | 每 Phase 结束 | 测试数量、工具数量等回填真实值 |

**Session 13 教训**：多次增量修改后，基线文档累积了 15 处格式/逻辑不一致。整个 Session 花在文档治理上。**预防方案**：每 5 个 Session 做一次全量审查。

## 五、人 + AI 协作模式

### 5.1 角色分工

| 维度 | 人类（决策者） | AI（执行者） |
|------|--------------|------------|
| **战略方向** | 定义 Phase 目标和优先级 | 评估可行性、提供方案选项 |
| **架构决策** | 最终拍板（如 SuperAgent vs Multi-Agent） | 分析利弊、推荐方案 |
| **代码实现** | 审查、验收 | 生成、修复、重构 |
| **质量守护** | 定义验收标准 | 执行测试、运行底线、修复 Bug |
| **知识沉淀** | 确认经验有效性 | 记录 logbook、更新基线、编码到 CLAUDE.md |
| **异常处理** | 判断是否放弃/换方向 | 最多 3 轮自修复后上报 |

### 5.2 高效协作模式

**模式 1：声明式意图 + 自主执行**

```
人类: "帮我实现 3 层记忆系统"
AI:   自主完成 → Entity 设计 → Repository → Service → Controller → 前端 UI → 测试 → 验收
人类: 审查结果 → 反馈调整
```

适用场景：功能明确、模式成熟的实现任务。Phase 5（记忆系统）1 个 Session 完成全栈实现。

**模式 2：探索式对话 + 渐进实现**

```
人类: "用户试用反馈了 4 个问题，我们需要重新规划 Phase 3"
AI:   分析反馈 → 提出方案 → 人类确认 → 逐步实现
人类: 每步审查 + 调整方向
```

适用场景：需求不明确、需要探索的场景。Session 21 的用户反馈→Phase 3 重规划就是典型案例。

**模式 3：并行 Agent 加速**

```
人类: "前端和后端可以并行做"
AI:   Agent A（后端） + Agent B（前端） → 并行执行 → 合并结果
```

适用场景：5+ 文件的独立功能集。Session 1 用 6 个 Agent 创建骨架，Session 29 用并行 Agent 拆分架构。

**并行 Agent 成功率**：

| Session | Agent 数 | 成功 | 部分成功 | 失败 | 成功率 |
|---------|---------|------|---------|------|--------|
| S1 | 6 | 2 | 3 | 1 | 33% |
| S2 | 2 | 2 | 0 | 0 | 100% |
| S4 | 2 | 2 | 0 | 0 | 100% |
| S29 | 2 | 2 | 0 | 0 | 100% |

教训：Session 1 的低成功率源于 Agent 权限不足和 rate limit。后续限制每次 2-3 个 Agent 并明确权限后，成功率 100%。

### 5.3 反模式

| 反模式 | 表现 | 代价 | 纠正方式 |
|--------|------|------|---------|
| **过度自动化** | AI 做了所有事但人不理解 | Session 21 用户反馈"黑盒感" | HITL 审批点 + 执行透明度 |
| **Docker 循环** | 改一行→重建 3 分钟→发现新问题 | Session 15 浪费 4 次重建 | 本地验证优先，Docker 仅做集成验证 |
| **后补验收** | 先写代码后补测试 | Session 13 发现 16 处错误 | 场景先行，编码前写预期 |
| **单入口测试** | 只测 REST API 不测 WebSocket | Session 14 workspace 工具 Bug | 同一入口多路径测试 |
| **大杂烩提交** | 一个 commit 包含 20+ 文件 | 难以 bisect 和回滚 | 每个逻辑变更一个 commit |

| 加速期 | S19-S28 | ~2,000 行 | 纪律内化后产出和质量同步提升 |
| 精炼期 | S29-S32 | ~1,200 行 | 重构为主，新代码减少但架构质量跃升 |

### 6.2 Session 类型分布

| 类型 | Session 数 | 占比 | 典型 Session |

**洞察**：Build 占 44% 是健康比例，但 Fix+Test 合计 28% 说明质量防线在早期不够强。从 Session 19 开始，验收测试前置后，后续 Session 的 Bug 修复比例明显下降。

## 七、方法论核心公式

### 7.1 交付质量公式

```
交付质量 = f(验收前置 × 底线拦截 × 基线守护 × 全局排查)
```

任何一项缺失都会导致质量下降：
- 缺验收前置 → Session 13 补测 16 处错误
- 缺底线拦截 → Session 5 CORS 通配符进 commit
- 缺基线守护 → 增量修改后文档逻辑矛盾
- 缺全局排查 → 枚举 Bug 重复出现 3 次

### 7.2 知识飞轮公式

```
交付 → 记录（logbook） → 验证（跨 Session 确认） → 编码（CLAUDE.md/Skill） → 更好的交付
  ↑                                                                            │
  └────────────────────────────────────────────────────────────────────────────┘
```

**当前状态**：飞轮的前 3 步（记录→验证→编码到 CLAUDE.md）已手动跑通并验证有效。第 4 步（编码为 Skill/Baseline 自动反哺）的框架已建立（LearningLoopPipeline），但**自动化链路尚未闭合**——这正是 Phase 8 的核心目标。

### 7.3 人 + AI 效率公式

```
效率 = AI 产出速度 × 人类决策质量 × 上下文连续性
```

- **AI 产出速度**：~1,700 行/Session，受限于 API rate limit 而非能力
- **人类决策质量**：通过验收测试和基线把控方向，避免"高速开往错误方向"
- **上下文连续性**：通过 logbook + CLAUDE.md + 3 层记忆保持，是三个因子中最脆弱的一个

## 九、度量指标

### 9.1 应持续跟踪的指标

| 指标 | 当前值 | 健康范围 | 说明 |
|------|--------|---------|------|
| 验收测试通过率 | 94.7% | ≥ 90% | 低于 90% 说明质量防线有漏洞 |
| Bug 修复率 | 96.7% | ≥ 95% | 低于 95% 说明有遗留问题积累 |
| Bug 复发率 | 0% | 0% | >0% 说明全局排查纪律未执行 |
| CLAUDE.md 增长率 | ~5 行/Session | 2-10 行 | 0 说明没在沉淀；>10 说明写太碎 |
| System Prompt 大小 | 20-25K chars | ≤ 30K | >30K 触发 rate limit |
| 首次部署成功率 | 100%（S14+） | ≥ 80% | <80% 说明 pre-flight 检查缺失 |
